# Twist.nixとは

[emacs-twist/twist.nix](https://github.com/emacs-twist/twist.nix)は、[Akirak](https://zenn.dev/akirak)さんによって書かれた、NixによるEmacsの管理ツールです。

Emacs本体とそのelisp packageおよび依存ツール（例えばtree-sitterなど）などをまとめて、[home-manager](https://github.com/nix-community/home-manager)のmoduleとして管理できます。

NixにはもちろんEmacsのpackageを管理するためのエコシステムがあるのですが、以下の点が不満足です。

1. Emacsの起動が遅くなりがち。
2. Packageの管理にNixを書かないといけない。

少なくとも、1に関してはtwist.nixは改善が見込めそうです。2に関しても多少はましだと個人的には思いました。基本的な設定さえ住んでしまえば、init.org（またはinit.el）に書けばいいだけなので。

日本語で書かれたよくまとまった紹介記事として、[Twist.nix で再現性のある Emacs 設定を作る](https://zenn.dev/kyre/articles/b1959567edfc15)があり、私はこれを読んで興味をもち使い始めました。ありがとうございます。

基本的には多少のNixとEmacsの知識がある方ならばこれを読めば使い始められると思うのですが、多少ハマったところがあったのでここに書き記して置こうと思います。なお、Nixに依存しているので残念ながらWindows nativeでは動作しません。わたしはWSL2とWSLgで使用しています。

# ハマったところ

閑話休題、ここからが本題です。

## setup.elの使用

わたしはuse-packageではなく[setup.el](https://codeberg.org/pkal/setup.el)を使っていたので、これをそのまま使おうと思いました。実際作者のakirakさんも同じようにつかっていたのでそれをしっかり見ておけばよかっただけなのですが、不注意でハマりました。

結論としては`initParser`と`extraPackages`の2つの設定が必要でした。

```nix: flake.nixの一部
inputs.twist.lib.makeEnv {
  initParser = inputs.twist.lib.parseSetup { inherit (inputs.nixpkgs) lib; } { };
  extraPackages = [ "setup" ];
}
```

`initParser`は多分defaultでは`use-package`なのだと思います（未確認）。

## LaTeX関連ツール

[AUCTeX](https://www.gnu.org/software/auctex/)と[pdf-tool](https://github.com/vedang/pdf-tools)のinstallに苦労しました。

最終的には`nix/override.nix`に依存するツールを定義して、それをflake.nixで読み込むことにより解決できました。

```nix: flake.nixの一部
pkgs.lib.composeExtensions inputs.twist-overrides.overlays.twistScope (
                import ./nix/overrides.nix { inherit pkgs; }
              )
```

```nix: nix/override.nix
{ pkgs }:

_tself: tsuper: {
  elispPackages = tsuper.elispPackages.overrideScope (
    _eself: esuper: {
      auctex = esuper.auctex.overrideAttrs (old: {
        outputs = [ "out" ];
        nativeBuildInputs = (old.nativeBuildInputs or [ ]) ++ [
          pkgs.perl
          pkgs.texliveSmall
        ];

        # GNUmakefileを書き換えてChangeLogの生成を避ける
        postPatch = (old.postPatch or "") + ''
          sed -i '/^ChangeLog:/,/^[^[:space:]]/c\
          ChangeLog:\n\
          \ttouch $@\n' GNUmakefile
        '';
        # homeless-shelter を避ける
        preBuild = (old.preBuild or "") + ''
          export HOME=$PWD/.home
          export XDG_CONFIG_HOME=$HOME/.config
          export XDG_CACHE_HOME=$HOME/.cache
          export XDG_DATA_HOME=$HOME/.local/share
          mkdir -p "$HOME" "$XDG_CONFIG_HOME" "$XDG_CACHE_HOME" "$XDG_DATA_HOME"
        '';

        # infoまで必要ならallにしないとだめ。必要ないなら make elpaでOK
        buildPhase = (old.buildPhase or "") + ''
          make all
        '';

        # infoは手動で移動しないとだめ
        installPhase = (old.installPhase or "") + ''
          mkdir -p $out/share/info
          if [ -d doc ]; then
              cp doc/*.info $out/share/info/ || true
          fi
          install-info $out/share/info/auctex.info $out/share/info/dir || true
          install-info $out/share/info/preview-latex.info $out/share/info/dir || true
        '';
      });

      pdf-tools = esuper.pdf-tools.overrideAttrs (old: {
        outputs = [ "out" ];
        nativeBuildInputs = (old.nativeBuildInputs or [ ]) ++ [
          pkgs.pkg-config
          pkgs.autoconf
          pkgs.automake
          pkgs.gcc
        ];
        buildInputs = (old.buildInputs or [ ]) ++ [
          pkgs.poppler
          pkgs.libpng
          pkgs.zlib
        ];
        buildPhase = (old.buildPhase or "") + ''
          runHook preBuild
          cd build/server
          autoreconf -i
          ./configure --prefix=$out --bindir=$out/bin
          make
          cd ../..
          runHook postBuild
        '';
      });
    }
  );
}
```

Nixとtwist.nixの書き方に慣れておらずどこに何を書けばいいのかわからなかったことと、依存ツールのNixでの名称がわからなかったことの2点のせいで苦労しました。GNUMakefileを書き換える部分では、ChatGPT等のサポートが強力でした。NixをつかうとAUCTeXの`make`を単純にすべて通すのは面倒そうだったのでこのような回避策を用いています。

またこの作業を通してoverlayについてなんとなく知ることができました。overlay関数は`final: prev: { ... }`という形で`prev`がoverlay適応前のpkgsセット、`final`が適応後のpkgsセットになります。したがって、上のコードでは`_tself: tsuper`は「上書き前と上書き後のパッケージ集合」、`_eself: esuper:`はその中のサブスコープ版というところでしょう。アンダースコア`_`は関数本体でつかっていないことを表しています。関数の本体で、各Phaseに介入できるようです。今回はPatchPhaseの直後 (PostPatch)とBuildPhaseの前 (preBuild) とBuildPhaseとInstallPhaseに介入しています（多い！）。依存関係の解決としてはnativeBuildInputsにはビルド時依存、buildInputsには実行時依存をいれればよいです。これを解決するのがとても面倒でした。毎回失敗させてそのたびにerror messageをみｔ1つずつ追加するという頭の悪い方法をとりましたが、絶対もっとなんかよいやり方があるのだとは思います。

この作業をするにあたり[§1. stdenv｜Nix入門: ハンズオン編](https://zenn.dev/asa1984/books/nix-hands-on/viewer/ch03-01-stdenv#buildinputs%E3%81%A8nativebuildinputs)が参考になりました。感謝します。

なお、`texliveSmall`（数百MB？）を依存に含むのは容量食いすぎではとも思ったのですが、Nix使う上では今更なので受けいれています。

## flake-utils vs flake-parts

そもそもvsで書くようなものでもない気がしますが、同時に使うことはなさそうなのでこういう書き方をしました。

現在は[numtide/flake-utils](https://github.com/numtide/flake-utils)]をつかっています。これは最初の方に学習した資料がflake-utilsを使っていたからというだけです。今となってはakirakさんやKyure-Aさんもつかっている[hercules-ci/flake-parts](https://github.com/hercules-ci/flake-parts)をつかっておけばよかったかなと思っています。簡潔に書くための機能がflake-partsの方に多そうです。
